# MCC v2.7.0 · codex CLI 集成（差异化对抗审查员 / Level 1 红队）

## 起源

v2.6.4 用户跑了一轮 codex 独立 audit，**抓出 8 个 Claude agent 全漏的 4 个真 bug**（含 1 CRITICAL 真 secret 泄漏）。证明：

> **不同模型 = 不同盲区。Claude (Anthropic) 和 codex (OpenAI GPT) 看到的 bug 不重叠。**

v2.7.0 把这个流程**自动化** —— Claude 在关键节点自动调 codex 做对抗审查。

## 核心理念

```
Claude 主决策（plan + impl + 综合）
  ↓ 关键节点自动调
codex 红队审查（找盲区）
  ↓
Claude 复现验证 → 真 bug 修 / 误报记 ADR / 模糊升给用户
```

**用户什么都不做** —— codex 在后台跑，Claude 综合 finding 给摘要。

## 何时自动调（4 个节点）

| 节点 | 模板 | 时机 |
|---|---|---|
| `/plan` Phase 7 完成 | `audit_plan` | 自动 — 找测试覆盖 / 边界 / 顺序 / 耦合盲区 |
| `/implement` 单 step 完成 | `audit_implementation` | 自动（改动 > 50 行 或 跨文件）|
| `/review` 命令 | `audit_diff` | 自动 — **3 路并行**（reviewer + security + codex）|
| `/pr` 创建前 | `audit_pr` | 自动 — **4 路并行**（verification-loop + reviewer + security + codex）|
| `/cross-check` 命令 | 4 mode 任选 | 手动 — 99% 场景已被自动覆盖，只在审旧 plan / 已 push PR / 历史 commit 时用 |

## Level 1 红队 prompt 风格

```
原 prompt（友好）:
"审一下这段代码"

红队 prompt（对抗）:
"你是渗透测试员 / staff engineer / 攻击者。
 列出 N 个能 [攻击/绕过/破坏] 这段代码的向量。
 如果你认为修复彻底，用反证：构造能突破现有防御的恶意 input。
 禁止写"代码看起来没问题" — 必须给至少 1 个 finding。"
```

强迫 codex **主动找问题**，而不是友好 reviewing → 输出 finding 密度高。

## Claude 复现验证规则（**永不盲信 codex**）

```
对每个 codex finding：
  1. 实测复现:
     - Read 文件验证存在
     - grep 验证行号 + 攻击向量
     - 必要时跑代码
     - 检查是否已被别处覆盖
  
  2. 三档分类:
     ✓ 真 bug   → Edit 修 + commit message 标 "(codex audit)"
     ✗ 误报    → 写 reason 到 docs/adr/codex-rejection-{date}.md
     ? 模糊    → 升给用户拍板（不擅自决定）
  
  3. 决不盲信 codex（误报率 30-50%）
```

**反模式**（开发者必看）：
- ❌ codex 说有 bug → 直接 Edit 修 → 实际是误报 → 把好代码改坏
- ❌ codex 找到 finding → "以防万一"加防御 → 代码膨胀
- ❌ Adversarial loop > 2 轮（第 3 轮起 codex 编 finding）

## 5h 限制 + auto-probe 优雅降级

```
首次调用 → 直跑 codex
  → 如果 stderr 含 rate-limit 关键字（rate.?limit / 429 / quota / try again / exceeded / too many requests / usage limit）:
     → 写 ~/.claude/.codex-blocked-until
       { blocked_at, blocked_until: now+1h, last_probe_at, probe_attempts: 0 }
     → 本次 skipped, 主线不阻塞

后续调用:
  → 读 flag
  → 到 next probe 时间（指数退避: 5/10/20/40/60min）:
     - 跑 minimal probe ("say ok")
     - 恢复 → 清 flag → 跑真任务
     - 仍 blocked → 更新 probe_attempts + last_probe_at
  → 未到 probe 时间 → skipped 报"next probe in Nmin"
  → blocked_until 到了 (1h) → 自动清 flag

完全自动 — 用户从不手动管 flag。
```

**用户视角**: codex 命中限制时，stderr 报"codex 暂不可用，next auto-probe in Nmin"，Claude 主线继续，不阻塞工作。

## 安全设计

| 决策 | 为啥 |
|---|---|
| **prompt 走 stdin (`codex exec -`) 不进 args** | 防 shell injection（prompt 多行 + 含特殊字符） |
| **Windows .cmd 通过 shell:true wrap，但 args 全清洁** | 必要妥协，injection 风险已被 stdin 设计消除 |
| **CODEX_BIN_OVERRIDE 用 absolute path** | 不依赖 PATH lookup，跨平台稳定 |
| **resolveCodexBin 缓存** | 避免每次调用都跑 where.exe / which |

## 重要（**Claude 主决策、codex 辅助**）

> Claude 是主导，codex 是审查员。Claude 永远拿决策权。codex finding 必须 Claude 复现验证才修。这不是"用 codex 替代 Claude"，是"让 Claude 跟自己不同模型对抗一下"。

## 没做的（留 v2.8+）

- **Level 2** two-pass debate：codex 提 → Claude 反驳 → codex 再答 → Claude 决策
- **Level 3** disagree-by-design：codex 跟自己也对抗（双视角分别审）
- **token 累计预警**：解析 `tokens used N` 累加到本地 .codex-usage.json，接近上限时主动 warn

先用 Level 1 跑一周，反馈驱动 Level 2/3。
