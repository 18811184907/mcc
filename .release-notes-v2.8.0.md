# MCC v2.8.0 · codex-runner 流式 streaming 重写

## 起源

用户问："**不能返回 codex 的思考过程吗？一定要等最终结果？类似于 codex 的窗口怎么实现的 cli 肯定不会 timeout 啊**"

正确——v2.7.x 用 `spawnSync` batch 模式，问题：
1. codex 大 prompt 思考 > timeoutMs 直接被砍
2. 没有实时进度，用户看像死了
3. Windows cmd /c stdin 走 GBK codepage 导致中文 prompt 乱码

v2.8 改 `spawn` + `--json` 流式读 jsonl 事件 → 跟 codex 桌面 / Web 同款体验。

## 核心改动

### `runCodexAudit` 是 async

```js
// v2.7.x (sync):
const r = runCodexAudit({ prompt, timeoutMs: 240_000 });

// v2.8 (async):
const r = await runCodexAudit({
  prompt,
  idleTimeoutMs: 90_000,    // 没新事件 90s 当卡死
  totalTimeoutMs: 600_000,  // 绝对上限 10min
  onEvent: (ev) => process.stderr.write(`[codex] ${ev.type}\n`),
});
```

**callers 必须改用 `await`**。skill / command 文档已更新。

### 流式事件类型（codex --json 输出）

```
thread.started       — 会话开始
turn.started         — 一轮思考开始
item.completed       — 输出一段（reasoning / agent_message / command_execution / tool_call）
turn.completed       — 含 usage.{input_tokens, output_tokens}
thread.completed     — 整个会话结束
```

**onEvent 回调**让 Claude 主线实时看到这些事件。可以打"心跳进度"给用户：

```
[3.3s]  thread.started
[3.3s]  turn.started
[23.0s] item.completed agent_message
[23.4s] item.started command_execution  ← codex 自己跑命令审 diff
[24.0s] item.completed command_execution
[33.8s] item.completed agent_message
...
```

### idle-timeout 替代 total-timeout

```js
// v2.7.x: 240s 没结果就杀（但 codex 在思考！）
// v2.8: 90s 没**新事件**才算 idle 杀。codex 在思考 = 还有事件流出 = 不算 idle。
```

idle-timeout 才是真正语义上的"卡死"。totalTimeoutMs 兜底（10 分钟绝对上限）。

### 实测真 codex（streaming 工作展示）

```
开始 streaming codex audit...
[3.3s]  thread.started
[3.3s]  turn.started
[23.0s] item.completed agent_message     ← 第 1 轮回答
[23.4s] item.started command_execution   ← codex 用 tool 看代码
[33.8s] item.completed agent_message     ← 第 2 轮
[47.3s] item.started command_execution
[66.4s] item.completed agent_message     ← 第 3 轮
[83.1s] item.completed agent_message     ← 第 4 轮
[93.8s] item.completed command_execution
---
SKIPPED: codex rate-limited just now (flag written, auto-probe will recover)
```

90+ 秒里清楚看到 codex **4 轮思考 + 多次 command_execution**。最终撞真 rate-limit（OpenAI 服务端限流，不是 false positive）—— **runner 正确识别**（exit 非 0 + stderr 含 rate-limit）。

### Windows stdin UTF-8 fix

```js
// v2.7.x: cmd /c <bin> exec - + spawnSync 的 input 字段 → 走 GBK codepage 中文乱码
// v2.8:   spawn + child.stdin.write(prompt, 'utf8') → 走 pipe 不走 cmd codepage
```

中文 prompt（红队模板含中文）现在直接以 UTF-8 喂 codex，不再乱码。

### child.on('exit') 替代 'close'

```js
// v2.7.x (close 等所有 stdio 关闭): SIGTERM 杀 cmd.exe 后 grandchild node 持有 pipe
//                                    → close 等 60s grandchild 自然 exit 才触发
// v2.8 (exit 触发于 process 退出):    cmd.exe 死了立刻 exit 触发 → finish() 强制
//                                    destroy stdio pipes → 立即 resolve

// 测试 idle-timeout 从 60s → 5.1s（实测）
```

### rate-limit 检测改进（删 "exceeded"）

v2.7.x 的关键词列表里 `exceeded` 太宽泛，codex 输出讨论"size exceeded"等正常话题误判。v2.8 删了它，只保留：

```js
RATE_LIMIT_PATTERNS = [
  /rate.?limit/i,
  /\b429\b/,
  /quota/i,
  /try again in/i,
  /too many requests/i,
  /usage limit/i,
];
```

仍配合 `(exit code 非 0) AND (stderr 含关键词)` 双校验。
