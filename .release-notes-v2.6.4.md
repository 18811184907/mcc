## 起源

v2.6.3 之后用户跑了一轮 **codex 模型独立 audit**——这是跟前两轮（普通 + opus）完全独立的第三轮交叉验证。codex 抓出 **1 CRITICAL + 3 HIGH + 2 MEDIUM**，全是前两轮**漏掉的真 bug**。

## 🔴 CRITICAL · 真 secret 泄漏路径

### `docs/PROJECT_VAULT.md` 不被 .gitignore 忽略

**问题路径**（codex 实测复现）：
1. v2.5.10 起 PROJECT_VAULT 默认路径从 `.claude/PROJECT_VAULT.md` 迁到 `docs/PROJECT_VAULT.md`
2. installer 创建新文件走 docs/ 路径（`scripts/installer.js:980`）
3. 但 `ensureProjectGitignore` 的 `required` 数组里**只有老路径** `.claude/PROJECT_VAULT.md`，缺 `docs/PROJECT_VAULT.md`
4. 老用户升级时 `.gitignore` 已有 `# MCC PROJECT_VAULT` marker → 旧 marker 检查直接 `return`，**升级也补不上**
5. 用户 commit 时 `git check-ignore docs/PROJECT_VAULT.md` 不命中 → secret 进 git 历史

**修复**（`scripts/installer.js`）：
1. `required` 加 `docs/PROJECT_VAULT.md`
2. 改 marker 检查逻辑：已有 marker 时**也检查 required 全在**，缺啥补啥（不再无脑跳过升级补漏）

**验证**（已实测）：
```bash
$ git check-ignore docs/PROJECT_VAULT.md
docs/PROJECT_VAULT.md         # ✓ 命中
```

升级后老用户 .gitignore 会被自动补一行：

```
✓ [project-stub] 🚫 .gitignore 补 1 行漏掉的 vault 路径（docs/PROJECT_VAULT.md）
```

**post-vault-sync.js 的 `ensureGitignore` 已经包含两条路径**——只 installer.js 那个漏了。两处共享 `required` 列表是 v2.6.5+ 重构候选。

## 🟠 HIGH × 3

### bootstrap.sh / .ps1 `rm -rf $MCC_DIR` 没路径校验

**问题**（`bootstrap.sh:96` + `bootstrap.ps1:74`）：git pull 失败时无脑 `rm -rf "$MCC_DIR"`。`MCC_DIR` 可由环境变量覆盖，**用户误设 `MCC_DIR=$HOME` 或别的目录，pull 失败就删错地方**。

**修复**：rm 前先校验 `$MCC_DIR/.git/config` 的 `remote.origin.url` 匹配 `18811184907/mcc` 或 `$REPO_URL`。不匹配 → 拒删 + 退出 + 提示手动检查：

```bash
case "$mcc_remote" in
  *18811184907/mcc*|*"$REPO_URL"*)
    rm -rf "$MCC_DIR"  # 真 MCC clone，可删
    ;;
  *)
    echo "[X] $MCC_DIR is not a recognized MCC clone (remote=$mcc_remote)"
    echo "[X] refusing to rm -rf — manually inspect: ls -la $MCC_DIR"
    exit 1
    ;;
esac
```

PowerShell 同款。

### `install.sh` 用 `$ROOT` 但未定义 → bash `set -u` 直接退

**问题**（`install.sh:76-77`）：`set -euo pipefail` 在第 14 行启用，但 `$ROOT` 全文从未定义。bash 进入到 line 76 就触发 `unbound variable` 错误退出，`install.sh` 完全跑不通。

**修复**：`$ROOT` → `$SCRIPT_DIR`（line 21 已有）。1 字之差 codex 静态分析抓到。

### `settings.fragment.strict.json` 缺 vault hooks

**问题**：选 `--strict` 模式的用户**反而失去 secret 防护** —— strict fragment 只有 SessionStart 和 Stop，没 PreToolUse `pre-vault-leak-detect` 也没 PostToolUse `post-vault-sync` / `post-user-vault-sync`。

**修复**：strict fragment 加齐 vault hooks。这跟 strict 哲学（最小信任、最严防护）一致——之前漏掉是设计 bug。

## 🟡 MEDIUM × 2

### `post-vault-sync.js` 写 `~/.ssh/config` 没走 secureWrite

**问题**：`~/.ssh/config` 直接 `fs.writeFileSync` 写入，没有 v2.6.2 在 user-vault-sync 实现的 symlink defense + atomic rename + Windows ACL lockdown。同项目里 user-vault-sync 走 secureWrite，project-vault-sync 没对齐。

**修复**：`syncSshConfig` + `syncSecretsIndex` 都改用 `secureWrite`。SECRETS-INDEX.md 是 committed 文件，传 `lockdownWindows: false` 跳过 icacls 但保留 symlink + atomic 防护。

### `gateguard-fact-force.js` MultiEdit 路径识别错位

**问题**（`gateguard-fact-force.js:382`）：MultiEdit 的 `tool_input` 结构是 `{file_path: "...", edits: [{old_string, new_string}, ...]}` —— `file_path` 在**顶层**，`edits[]` 只有 old/new strings。但 hook 从 `edit.file_path` 取（永远是 `undefined`）→ 等价 MultiEdit 全 allow（绕过 Edit 的 gate）。

**codex 实测**：
> `Edit` 会 deny，等价 `MultiEdit` 被 allow

**修复**：从 `toolInput.file_path` 顶层取，保留 `edits[].file_path` 作 belt-and-suspenders fallback（兼容 schema 变动）。
