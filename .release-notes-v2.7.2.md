# MCC v2.7.2 · probeRecovery 同款 false positive hotfix（Layer 2 抓到）

## 起源（meta-dogfood 故事）

刚发完 v2.7.1（Layer 2 二审 + auto-fix），用户实测说 **codex 还在被报限流**。我答"等 codex 恢复"。用户回："**我刚刚测试了 codex 没有到限额 你为什么要等 codex**"。

用户给了 ChatGPT dashboard 截图：**91% 剩余**。

矛盾：codex 真没限流，但我的 runner 报限流。**第二个 false positive bug**——v2.7.1 修主路径时漏掉了 probeRecovery 函数。

正好测试 Layer 2 链路时我派了 finding-validator subagent 反向审 v2.7.1（用一个已知 bug 的描述让 validator 复现）。**结果 validator 给 confirmed verdict** —— 不是因为我描述的那个 bug 还存在（那个修了），**是因为 validator 在 Read 文件时主动发现了 probeRecovery 函数有同款未修 bug**。

```
Layer 2 finding-validator 输出原话:
  VERDICT: confirmed
  REASONING: codex 描述的 bug 在主路径已修复。但 probeRecovery 函数（line 191-196）
              是另一个独立代码路径，仍然使用 combined = stderr + '\n' + stdout
              然后扫合并字符串——同样的假阳性 bug 在这里没有被修复。
  FIX_GUIDANCE: [完整 unified diff]
```

按 v2.7.1 设计：**confirmed → Layer 3 主 Claude 自动 Edit 修不询问用户**。我立即按 fix_guidance 修了。

## 这次 dogfood 的真正意义

**v2.7.1 三层架构在 v2.7.1 自己身上 work 了**：
- Layer 1（codex 提 finding）：本来要 codex 来报，但 codex 限流。所以我用 mock finding 派 validator 测链路
- Layer 2（finding-validator subagent 二审）：fresh subagent 不带我对 v2.7.1 修复的"已经修好了"信念，独立 Read 文件 → **抓出我自己漏修的 probeRecovery**
- Layer 3（主 Claude 自动修）：按 validator confirmed verdict 的 fix_guidance 直接 Edit，没问用户

**结果**：用户实测发现 v2.7.0 的 false positive，v2.7.1 修了主路径，**Layer 2 设计抓出 v2.7.1 自己漏的同款 bug**，v2.7.2 自动修。

这就是"差异化盲区"——单 LLM 一次性修复不会 100% 完整，**外部反馈循环（用户 + 跨 agent 审查）才能真正捞干净**。

## 修复

```js
// v2.7.1 (probeRecovery 仍有 bug):
const combined = stderr + '\n' + stdout;
if (RATE_LIMIT_PATTERNS.some(re => re.test(combined))) {
  return true; // ← 误判
}

// v2.7.2 (与 runCodexAudit 主路径一致):
if ((result.error || result.status !== 0)
    && RATE_LIMIT_PATTERNS.some(re => re.test(stderr))) {
  return true;
}
```

**实测**：清 stale flag + 跑真 codex（91% 剩余）→ `SKIPPED: false, output: 'ok', tokens: 11884` ✓
