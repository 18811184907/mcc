### 🔄 D2: user-vault 默认改 PROJECT（行为变化）

**之前**（v2.6.0-2.6.2）：拿不准时 Claude 默认 USER。
**v2.6.3**：拿不准时默认 PROJECT。

理由——**写错 USER 比写错 PROJECT 严重得多**：

| 场景 | 后果 |
|---|---|
| 实际是 PROJECT 但写到 USER | 全机污染 + 跨项目可能用错 secret + 难恢复 |
| 实际是 USER 但写到 PROJECT | 多写一次（下个项目重填）—— 易恢复 |

只在用户**明确说**"我的 / 个人 / 跨项目"时才走 USER。其他模糊场景默认 PROJECT。

### 📖 D1: "PROJECT 覆盖 USER" 文案诚实化

之前 SKILL.md 说 "PROJECT 覆盖 USER"——**这只在 Node + dotenv (默认 `override:true`) 项目里成立**。其他工具栈实际是 USER 赢（shell env 兜底）。新增工具栈对照表：

| 工具栈 | 同名 key 谁赢 |
|---|---|
| Node + dotenv 默认 (`override:true`) | **PROJECT 赢** |
| Node + dotenv `override:false` | USER 赢 |
| Python `os.environ` 直接读 | **USER 赢** |
| Python + python-dotenv 默认 | USER 赢 |
| Go / Rust / Shell 脚本 | **USER 赢** |
| Vite / Next.js (server-side) | dotenv 系，PROJECT 赢 |

**实务建议**：避免 USER 和 PROJECT 同名 key（容易混淆）。

### 🌐 D3: 跨机同步 3 方案 FAQ

之前 SKILL.md 只说"用密码管理器或手动 scp"——v2.6.3 扩写 3 方案：

**方案 1（推荐）**：**密码管理器** + `MCC_NO_AUTOLOAD=1` opt-out
```bash
# ~/.bashrc
export MCC_NO_AUTOLOAD=1
export OPENAI_API_KEY=$(op read 'op://Personal/OpenAI/api-key')
export ANTHROPIC_API_KEY=$(op read 'op://Personal/Anthropic/api-key')
```
secret 在密码管理器，shell 启动注入。MCC 不动 profile。最安全。

**方案 2**：**dotfiles repo + SOPS / age 加密**
```bash
sops -e USER_VAULT.md > USER_VAULT.md.enc
git add USER_VAULT.md.enc && git commit
# 新机:
sops -d USER_VAULT.md.enc > ~/.claude/USER_VAULT.md
```
解密后让 Claude 触发 user-vault-sync。

**方案 3（最简）**：`scp ~/.claude/USER_VAULT.md newmachine:~/.claude/`，新机 Claude `Edit` 触发 hook 重建 .user-env.{sh,ps1}。

**反模式**：`USER_VAULT.md` 直接 push 到 git（即使 private repo）—— git history 留 secret 痕迹永远擦不掉。要走 git **必须用方案 2 加密**。
